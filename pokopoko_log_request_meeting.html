📅 2026.04 📋 PM팀 내부 검토용
배경 및 요청 목적

결과를 넘어, 선택의 맥락을 보기 위한
로그 추가 요청입니다

지금까지의 분석으로 난이도, 이탈, continue 사용, 아이템 사용 같은 결과 지표는 어느 정도 볼 수 있게 되었습니다. 이번 요청은 그 결과를 대체하기 위한 것이 아닙니다. 유저가 어떤 상태에서 그 선택을 했는지, 플레이 중 어느 구간에서 압박이 생겼는지를 보강하기 위한 요청입니다.

현재 볼 수 있는 것
결과 지표 중심
어떤 스테이지에서 많이 막히는지
continue 및 아이템 사용량이 어디서 높은지
이탈이 어느 구간에서 커지는지
스테이지 클리어율 및 패배율
지금 부족한 것
선택 직전의 상태와 맥락
continue 사용 시점에 목표가 얼마나 남았는지
아이템을 어떤 상황(턴, 목표 잔량)에서 꺼내는지
플레이 중 어느 구간부터 진행이 정체되는지
near-miss 상황의 경향성 (일반적으로 알려진 패턴이 우리 게임에도 적용되는지)
이탈의 흐름
플레이 중 압박 발생
→
스트레스 누적
→
continue / 아이템 선택
→
⚠ 이탈 (결과)

이탈은 스트레스가 충분히 쌓인 이후의 결과입니다. 장기 서비스 운영에서 중요한 것은 이탈 그 자체보다, 그 직전에 나타나는 압박 신호와 선택의 맥락을 먼저 포착하는 것입니다.

LOG 01
continue_goal
신규 로그
continue 사용 시점의 남은 목표 수와 몇 번째 continue인지 기록. 어떤 상황에서 continue를 선택하는지, near-miss 경향성이 실제로 존재하는지 파악 가능.
LOG 02
ingame_item 사용 맥락
기존 로그 보완
아이템 사용 시점의 남은 턴 수와 남은 목표 수 추가. 유저가 어떤 압박 상황에서 아이템을 꺼내는지 행동 패턴 파악 및 아이템 사용 유도 설계에 활용 가능.
LOG 03
진행도 분석 로그
신규 로그
시작 턴을 1/n 구간으로 나눠 각 구간마다 남은 목표 수·남은 턴 수 기록. 어느 구간에서 진행이 정체되고 압박이 커지는지 흐름으로 파악 가능.

세 로그의 공통 목적 — 설계 의사결정의 근거 확보

📊
이탈 선행 신호 포착 → 위험 스테이지 사전 감지 이탈은 이미 스트레스가 누적된 결과입니다. 이 로그들은 이탈 이전에 나타나는 압박 신호를 포착하여, 이탈률이 쌓이기 전에 위험 스테이지를 빠르게 식별하고 수정하기 위한 근거를 제공합니다.
🎯
유저 선택 맥락 확보 → 난이도·아이템·컨티뉴 설계 의사결정 변경 continue와 아이템을 어떤 상황에서 사용하는지를 알면, 난이도 조정 기준 명확화, 아이템이 필요해지는 구간의 의도적 설계, 스테이지 내 병목 구간 특정 후 직접 수정 등 설계 의사결정을 실제 데이터에 근거하여 내릴 수 있게 됩니다.
⚙️
장기 운영 자동 모니터링 기반 구축 세 로그 모두 PLAYEND 상시 로그에 붙는 구조입니다. 신규 챕터 오픈 시 초기 데이터만으로 위험 패턴을 자동 감지하고, 스트레스 지수 프레임워크 및 DDA 시스템의 트리거 기준을 설계하는 데 기초 데이터로 활용됩니다.

continue_goal

continue 사용 시점에 남은 목표 수와 몇 번째 continue인지 기록하는 신규 로그입니다. PLAYEND 상시 로그에 포함되는 구조이며, 기존 L_CNTNE_ITEM에 현재 담기지 않는 "어떤 상태에서 눌렀는지"를 보완합니다.

신규 로그 PLAYEND 상시
현재 가능한 것
  • 스테이지별 continue 사용 횟수 집계
  • continue 사용 여부 (L_IREP 필드)
  • continue 시 코인 잔액 (L_1ST, L_2ND)
  • 몇 번째 continue인지 (L_CIID)
현재 부족한 것
  • continue를 누를 때 목표가 얼마나 남아 있었는지 알 수 없음
  • 거의 클리어 직전이어서 눌렀는지, 한참 남은 상태에서 압박이 커서 눌렀는지 구분 불가
  • 보편적 3매치에서 알려진 near-miss 패턴이 포코포코에서도 나타나는지 확인 불가
  • continue의 "선택 맥락"이 없어 행동 해석에 한계가 있음
보완되는 것
  • continue 시점의 남은 목표 수를 알 수 있어 "거의 다 왔을 때 쓰는지" 파악 가능
  • near-miss 행동 패턴 존재 여부를 데이터로 처음으로 확인 가능
  • 2~3번째 continue 사용 시의 목표 잔량 변화도 추적 가능
  • continue 선택의 맥락이 쌓이면 스트레스 지수 정교화에 활용 가능
활용 방향 — 구체적 액션
  • near-miss 비율이 높은 스테이지 → 난이도 미세 조정 근거 확보. 목표 1~2개 남은 상태에서 continue 비율이 유난히 높다면, 유저가 "아깝다"고 느끼는 스테이지. 턴 수 소폭 조정이나 후반 블록 구성 완화 등 "억울한 실패" 감정을 줄이는 설계 판단이 가능해짐
  • continue 시점 목표 잔량 패턴 → 난이도별 컨티뉴 선택 경향성 비교. 목표가 많이 남은 상태에서도 continue를 누르는지, near-miss에서만 집중되는지에 따라 난이도별 경향성이 달라짐. 컨티뉴 추가 턴 수의 적정성을 검토하는 근거
  • 이탈 스테이지의 continue 목표 잔량 분포 → 이탈 선행 신호 포착. "목표 많이 남은 채 continue → 실패 → 이탈" 패턴이면 난이도 자체가 과한 것, "near-miss 반복 후 이탈"이면 억울함 누적. 같은 이탈이어도 조정 방향이 달라지는 근거
  • continue 깊이 + 목표 잔량 조합 → 스테이지별 위험도 자동 판별 기준 수립. 스트레스 지수 프레임워크에 continue 관련 축을 정량화하여, 위험 스테이지를 자동으로 식별하는 모니터링 기준으로 활용
샘플 포맷 (PLAYEND 내 포함 구조)
// PLAYEND 로그 내 continue_goal 필드 구조 예시 { "continue_goal": { "continue_seq": 2, // 몇 번째 continue인지 "goal_remain": [ { "goal_type": "crack", // 목표 유형 (석판, 열쇠 등) "remain": 3 // 남은 개수 }, { "goal_type": "key", "remain": 1 } ] } } // 구조 설계 참고: PLAYEND 안에 들어가는 구조이므로, seq와 goal이 분리된 평면 구조보다 // 하나의 덩어리로 묶인 형태가 적절합니다. (기존 L_CHAR, L_CNTNE_ITEM 구조와 동일한 방식)

ingame_item 사용 맥락 추가

기존 ingame_item 사용 로그에 남은 턴 수와 남은 목표 수를 추가하는 보완 요청입니다. 아이템 사용량은 볼 수 있지만, "어떤 상황에서 꺼냈는지"가 빠진 지금의 한계를 해소합니다.

기존 로그 보완 PLAYEND 상시
현재 가능한 것
  • 스테이지별 ingame_item 사용 여부
  • 어떤 아이템 종류를 사용했는지
  • 사용한 아이템의 개수
  • 아이템 사용이 높은 스테이지 파악
현재 부족한 것
  • 아이템을 몇 턴 남았을 때 사용했는지 알 수 없음
  • 아이템 사용 시 목표가 얼마나 남아 있었는지 알 수 없음
  • "어려운 상황에서 쓰는지" vs "여유 있을 때 쓰는지" 구분 불가
  • 아이템 사용 유도 설계 시 근거가 될 맥락 데이터 부재
보완되는 것
  • 아이템 사용 시점의 턴 잔량과 목표 잔량을 함께 볼 수 있게 됨
  • 아이템별 사용 경향성 비교 가능 (bombswitch vs crosshammer 등)
  • 고압박 구간(턴·목표 모두 촉박)에서 아이템 사용 패턴 분석 가능
  • 아이템 효과와 실제 클리어 여부의 관계 해석 강화
활용 방향 — 구체적 액션
  • 아이템 사용 시점(턴·목표 잔량) 패턴 → 아이템 필요 구간을 의도적으로 설계하는 근거. 예: crosshammer가 "턴 3 이하 + 목표 5 이상"에서 집중 사용된다면, 스테이지 설계 시 해당 구간의 보드 구성에서 아이템 필요성을 체감하게 만드는 설계가 가능해짐
  • 압박 상황에서 아이템 미사용 / 써도 효과 없는 스테이지 식별 → 난이도 조정 대상 우선순위 판단. 턴·목표 모두 촉박한데 아이템을 안 쓰거나, 써도 클리어 전환이 안 되는 스테이지는 아이템 효용이 낮은 보드 구성일 가능성이 높음. 난이도 자체를 손봐야 하는 대상으로 우선 선별 가능
  • 아이템 종류별 사용 맥락 비교를 통해, 스테이지 보드 구성 시 어떤 아이템이 효과적으로 작동하는 구조인지 참고 가능
  • item_Clear_Lift를 압박 구간별로 세분화하면, 기존에 스테이지 단위로만 보던 아이템 효용 지표의 해석력이 강화됨
샘플 포맷 (PLAYEND 내 포함 구조)
// PLAYEND 로그 내 ingame_item_usage 필드 구조 예시 { "ingame_item_usage": [ { "item_type": "crosshammer", // 아이템 종류 "remain_turn": 4, // 사용 시점의 남은 턴 수 "goal_remain": [ { "goal_type": "crack", "remain": 6 } ] }, { "item_type": "bombswitch", "remain_turn": 2, "goal_remain": [ { "goal_type": "crack", "remain": 2 } ] } ] } // 한 플레이 내에서 여러 아이템을 사용한 경우, 배열로 순서대로 기록합니다.

진행도 분석 로그

시작 턴을 1/n 구간으로 나누어 각 구간마다 남은 목표 수와 남은 턴 수를 기록하는 신규 로그입니다. 플레이 중 어느 구간에서 진행이 정체되고 압박이 커지는지를 흐름으로 파악할 수 있습니다. n 값은 개발 논의가 필요합니다.

신규 로그 PLAYEND 상시
현재 가능한 것
  • 플레이 결과(클리어/실패) 확인
  • 최종 점수, 플레이 시간
  • 아이템·continue 사용 여부 (결과 시점)
  • 사용한 레디아이템 목록
현재 부족한 것
  • 플레이 중간의 진행 상태를 전혀 볼 수 없음
  • 어느 구간에서 목표 감소가 정체됐는지 알 수 없음
  • 후반부에 급격히 압박이 커졌는지, 처음부터 막혔는지 구분 불가
보완되는 것
  • 플레이 흐름을 구간별로 추적 가능 (1/4, 2/4, 3/4, 마지막 구간)
  • 어느 구간에서 목표 감소 속도가 급격히 떨어지는지 분석 가능
  • progress_ratio 기준으로 스테이지 간 비교, remain_turn은 실제 압박 흐름 참고용으로 병행 활용
  • 스테이지 유형별 진행 압박 패턴 비교 분석 기반 확보
활용 방향 — 구체적 액션
  • 구간별 목표 감소율 패턴 → 병목 구간 특정 → 해당 구간 보드 구성 직접 수정. 75% 구간에서 목표 감소율이 급락하면 후반 보드를 손보면 되고, 처음부터 정체면 초반 블록 배치를 조정. "이 스테이지가 어렵다"를 넘어 "어디서부터 막히는지"를 알아 난이도 수정 방향이 특정됨
  • 이탈 스테이지의 진행도 곡선 공통 패턴 추출 → 위험 스테이지 사전 감지 자동화. 신규 챕터 오픈 후 이탈률이 쌓일 때까지 기다리지 않고, 초기 진행도 패턴만으로 "이 스테이지 위험하다"를 빠르게 포착. 이탈의 선행 신호로 활용
  • 구간별 진행 정체도를 스트레스 지수 프레임워크에 축으로 추가. 현재 2축 뷰(스트레스 + 구조 효율)에서 스트레스 축의 "board lock" 관련 지표를 구간별 진행 정체도로 정밀하게 보완. 스테이지 간 비교 기준이 더 정교해짐
  • DDA / 유저 난이도 개인화 시스템의 트리거 조건 설계 시 기초 데이터로 활용. 어느 구간에서 압박이 커지는지의 패턴 데이터가 있어야 동적 난이도 조정의 개입 시점과 강도를 설계할 수 있음
샘플 포맷 (n=4 기준, PLAYEND 내 포함 구조)
// PLAYEND 로그 내 progress_goal 필드 구조 예시 (n=4 구간 기준) // remain_turn을 함께 포함하는 이유: ready_item 사용으로 시작 턴이 달라질 수 있어 // 구간이 같더라도 실제 남은 턴 수가 다를 수 있기 때문 { "progress_goal": [ { "progress_step": 1, "progress_ratio": 0.25, // 전체 턴의 25% 시점 "remain_turn": 18, "goal_remain": [ { "goal_type": "crack", "remain": 18 } ] }, { "progress_step": 2, "progress_ratio": 0.50, "remain_turn": 12, "goal_remain": [ { "goal_type": "crack", "remain": 11 } ] }, { "progress_step": 3, "progress_ratio": 0.75, "remain_turn": 6, "goal_remain": [ { "goal_type": "crack", "remain": 8 } ] }, { "progress_step": 4, "progress_ratio": 1.00, // 마지막 구간 (플레이 종료 시점) "remain_turn": 1, "goal_remain": [ { "goal_type": "crack", "remain": 5 } ] } ] } // n 값 (구간 분할 수): 개발 논의 필요. 데이터 부담과 분석 정밀도 사이의 균형 고려.
함께 고려할 수 있는 추가 항목

아래 두 항목은 진행도 로그와 같은 맥락에서 함께 적재하면 분석 활용도를 높일 수 있는 내용입니다. 필수는 아니며, 개발 부담을 고려해 선택적으로 검토하는 방향을 제안드립니다.
